## Description

Define and implement the standard CI workflow for individual slide repositories: compile to `wasm32-wasip1`, run tests, package the result as a `.vzglyd` archive, and attach it to a GitHub Release. This makes every official slide downloadable as a single file that users can drop into their slides directory.

## Background

The `.vzglyd` package format (established in E4) is a zip archive containing:
- `slide.wasm` — the compiled WASM binary
- `manifest.json` — metadata (name, abi_version, scene_space, display config)
- `shaders/` — WGSL shader files (if any)
- `assets/` — textures, models, embedded data (if any)
- `sidecar` — compiled sidecar binary (WASI executable, if applicable)

Users install a slide by copying its `.vzglyd` file into `/var/lib/vzglyd/slides/`. This ticket standardises the CI that produces that file.

## Standard slide CI workflow

Every slide repo under `vzglyd/` gets a `.github/workflows/ci.yml` following this template:

```yaml
name: CI

on:
  push:
    branches: [main]
    tags: ["v*"]
  pull_request:
    branches: [main]

env:
  RUST_BACKTRACE: 1

jobs:
  test:
    name: Test (host)
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
      - run: cargo test

  build-wasm:
    name: Build WASM
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
        with:
          targets: wasm32-wasip1
      - name: Build slide WASM
        run: cargo build --target wasm32-wasip1 --release
      - name: Build sidecar WASM
        if: ${{ hashFiles('sidecar/Cargo.toml') != '' }}
        run: cargo build --manifest-path sidecar/Cargo.toml --target wasm32-wasip1 --release
      - name: Package .vzglyd
        run: bash build.sh
      - name: Upload .vzglyd artifact
        uses: actions/upload-artifact@v4
        with:
          name: slide-package
          path: "*.vzglyd"

  release:
    name: GitHub Release
    if: startsWith(github.ref, 'refs/tags/v')
    needs: [test, build-wasm]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/download-artifact@v4
        with:
          name: slide-package
      - uses: softprops/action-gh-release@v2
        with:
          files: "*.vzglyd"
          generate_release_notes: true
```

## Build script contract

Each slide's `build.sh` must produce a `<slide-name>.vzglyd` file at the repo root. The standard `build.sh` (generated by the cargo-generate template):

```bash
#!/usr/bin/env bash
set -euo pipefail
cd "$(dirname "$0")"

SLIDE_NAME=$(cargo metadata --no-deps --format-version 1 \
    | jq -r '.packages[0].name' | sed 's/_slide$//')

# Build slide WASM
cargo build --target wasm32-wasip1 --release

# Build sidecar WASM (if present)
if [[ -f sidecar/Cargo.toml ]]; then
    cargo build \
        --manifest-path sidecar/Cargo.toml \
        --target wasm32-wasip1 \
        --release
fi

# Package
TMP=$(mktemp -d)
trap 'rm -rf "$TMP"' EXIT

mkdir -p "$TMP/pkg"
cp manifest.json "$TMP/pkg/"

# WASM: find by package name
cp "target/wasm32-wasip1/release/${SLIDE_NAME}_slide.wasm" "$TMP/pkg/slide.wasm"

# Sidecar
if [[ -f sidecar/Cargo.toml ]]; then
    SIDECAR_NAME=$(cargo metadata --manifest-path sidecar/Cargo.toml \
        --no-deps --format-version 1 | jq -r '.packages[0].name')
    cp "sidecar/target/wasm32-wasip1/release/${SIDECAR_NAME}.wasm" "$TMP/pkg/sidecar"
fi

# Shaders
if [[ -d shaders ]]; then
    cp -r shaders "$TMP/pkg/"
fi

# Assets
if [[ -d assets && -n "$(ls -A assets)" ]]; then
    cp -r assets "$TMP/pkg/"
fi

# Create archive
(cd "$TMP/pkg" && zip -r "$OLDPWD/${SLIDE_NAME}.vzglyd" .)
echo "Packaged: ${SLIDE_NAME}.vzglyd"
ls -lh "${SLIDE_NAME}.vzglyd"
```

## Versioning convention for slides

Slides follow semver independently of the engine. Version is declared in `manifest.json`:

```json
{
  "name": "weather",
  "version": "0.2.1",
  "abi_version": 1,
  ...
}
```

The GitHub Release tag matches: `v0.2.1`. The `.vzglyd` filename includes the version: `weather-0.2.1.vzglyd` — this allows users and the registry to pin specific versions.

**Update build.sh to include version in filename:**

```bash
VERSION=$(jq -r .version manifest.json)
OUTFILE="${SLIDE_NAME}-${VERSION}.vzglyd"
```

## Slide compatibility declaration

The `manifest.json` `abi_version` field is the engine compatibility signal. The release notes for each slide version should mention which engine versions are compatible.

For the registry (E12-T10), each slide entry will include a `min_engine_version` field that the registry derives from the slide's `abi_version`.
